# ConvoMatrix · 复杂指令下的多轮对话评测系统

**DialogEval — 让 AI 从「会聊天」，走向「能办事」**

---

| 项目信息 | |
|---|---|
| 产品名称 | ConvoMatrix（系统代号 DialogEval） |
| 项目类型 | 多轮对话 AI / 数字人自动评测平台 |
| 核心能力 | 将任意复杂任务指令，自动映射为统一五维评分标准并产出可解释报告 |
| 技术栈 | FastAPI · React · DeepSeek · SSE 流式对话 |

---

## 一、我们解决什么问题？

在企业客服、外呼通知、销售跟进等场景中，对话式 AI 的上线门槛不在「会不会说话」，而在**能不能按业务指令把事办成**。

然而，业务方编写的任务指令往往：

- **格式各异**：有的写 `# Call Flow`，有的写 `# Conversation Flow`，字段命名、层级结构都不统一；
- **逻辑复杂**：同一流程里混杂「必须主动完成的步骤」「若用户拒绝则挽留」「被问及才回复的话术」；
- **内容多样**：FAQ 知识库、占位符变量（X 单、Y 天）、字数/语气/禁止词等约束并存；
- **路径分叉**：配合型、拒绝型、质疑型、终止型用户，同一条指令在不同分支下应有不同评分预期。

**传统做法**是为每条指令人工编写评测 rubric，成本高、标准难统一、难以横向对比不同 Agent 版本。

**ConvoMatrix 的做法**：用户只需粘贴任务指令，系统自动完成「解析 → 多分支模拟 → 统一标准打分 → 可视化报告」，把**不规则的业务语言**翻译成**可度量、可对比、可审计的评测结果**。

## 二、核心创新：一套标准，适配万千指令

### 2.1 创新点概述

ConvoMatrix 的独特优势，不在于「又一个对话 Demo」，而在于建立了一套**指令无关（Instruction-Agnostic）的统一评测框架**：

> **无论上游指令多么复杂、结构多么不统一，最终都汇入同一套五维评分模型；**
> **无论评分点语义多么不同，都通过分类评分逻辑公平、准确地执行。**

这意味着：

| 对比维度 | 传统人工评测 | ConvoMatrix |
|----------|-------------|-------------|
| 新指令接入 | 重新写 rubric，数天级 | 粘贴即解析，分钟级 |
| 评分标准 | 因评测员/场景而异 | 固定五维权重，可横向对比 |
| 条件分支 | 难以界定「该不该评」 | 分支感知 + 场景触发检测 |
| FAQ / 占位符 | 易误扣或漏评 | 语义类型分流，按场景评分 |
| 结果解释 | 主观描述 | 每个子项附带 ≤60 字中文理由 |

### 2.2 统一五维评分框架

所有任务指令，无论来源行业、格式风格，最终都映射为以下五个维度：

| 维度 | 权重 | 评估内容 |
|------|------|----------|
| **任务完成度** | 35% | 核心目标是否达成、流程步骤是否执行、FAQ 是否答准 |
| **指令遵循** | 25% | 字数限制、禁止词、避免重复等硬性约束 |
| **自然度** | 15% | 口语化程度、电话沟通感、是否符合角色设定 |
| **分支处理** | 15% | 条件场景下的应对是否恰当（挽留、超范围回复等） |
| **效率** | 10% | 是否在合理轮次内完成任务 |

**评委可验证的价值**：不同业务、不同指令，报告页始终是同一套雷达图结构——适合 A/B 测试、版本迭代追踪与质量门禁。

### 2.3 语义类型分流：复杂逻辑也能评准

任务指令中的内容，语义上至少分为五类。ConvoMatrix 为每个评分点标注 **`item_kind`**，按类型采用不同评分逻辑，避免「一刀切」导致的误评：

| 语义类型 | 典型来源 | 评分原则 |
|----------|----------|----------|
| **主动步骤** `mandatory_step` | Call Flow 中 Agent 必须发起的节点 | 未完成即扣分，不存在「场景未触发就满分」 |
| **条件响应** `conditional_response` | 「若拒绝则挽留」「如被问及则回复…」 | 仅当用户触发该场景才评分；未触发 → 标记「不适用」 |
| **FAQ 条目** `faq_entry` | Knowledge Points | 仅当用户问到相关话题才评分；未问 → 不适用 |
| **全程约束** `constraint` | 字数、禁止词 | 确定性规则通道，可复现 |
| **开场白** `opening` | Opening Line | 语义覆盖核心信息；占位符可用具体数字替代 |

**举例**：美团飞毛腿场景中，Call Flow 第 3 步「挽留不想配送 / 鼓励能配送 / 提醒安全」会被**自动拆成 3 个独立评分点**——挽留只评拒绝型分支，鼓励只评配合型分支，提醒安全在所有分支都要评。这是人工 rubric 极易遗漏、而 ConvoMatrix 通过 Parser + 后处理归一化自动完成的。

### 2.4 三种评估通道：确定性 + 语义，各取所长

每个评分点还标注 **`eval_type`**，运行时走最适合的通道：

| 通道 | 适用场景 | 特点 |
|------|----------|------|
| **规则 `rule`** | 每轮字数、禁止词、重复检测、开场关键词 | 零幻觉、可复现 |
| **关键词 `keyword`** | 简单无条件的要点命中 | 快速、透明 |
| **语义 `llm`** | 步骤完成度、FAQ 准确性、条件应对、占位符等价 | 理解上下文与语义 |

Parser 输出后，系统还会执行**确定性后处理**（`criteria_normalize`），自动修正常见误分类：条件句强制走语义评分、FAQ 禁止字面关键词匹配、含 X/Y/Z 占位符改语义等价判定、并列对立策略自动拆项等。

**创新本质**：LLM 负责「理解不规则指令」，规则引擎负责「守住可量化底线」，二者在统一框架下协同——既灵活又可控。

### 2.5 多分支覆盖真实用户路径

系统从指令中自动生成 **2–4 个用户分支**（配合型 / 拒绝型 / 质疑型 / 终止型等），每个分支由独立的 User 模拟器 persona 驱动。

条件性子项通过 **`applicable_branches`** 限定只在会触发该场景的分支上评分，其余分支标记「不适用」，不计入维度均分。

**价值**：不只测「理想用户下的表现」，还测边缘场景——这正是对话 AI 上线前最缺、又最难人工覆盖的部分。

## 三、产品工作流

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  粘贴指令   │ →  │  智能解析   │ →  │  分支模拟   │ →  │  自动评分   │
│  (任意格式) │    │  分支+标准  │    │  双LLM对话  │    │  五维报告   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

### Step 1 · 输入任务指令

支持粘贴任意业务指令，最少包含 `# Role`、`# Task`、`# Opening Line`、`# Constraints`；可选 `# Call Flow`、`# Knowledge Points (FAQ)` 等，格式不完全统一也可解析。

内置两个高复杂度预置样例，可直接 Demo：

1. **美团外卖·飞毛腿骑手通知** — 外呼 SOP + FAQ + 条件挽留/鼓励 + 占位符变量 + 超范围回复约束  
2. **课程平台·直播升级客服** — 7 步 Conversation Flow + 嵌套条件分支 + 特殊终止场景（商家在开车）

### Step 2 · 指令解析

LLM Parser 将自然语言指令结构化为：

- 用户分支列表（含 persona 剧本）
- 五维评分标准（含 20+ 可审计子项）
- 语气摘要

解析结果可在分支页预览，透明展示「系统如何理解你的指令」。

### Step 3 · 流式对话模拟

- **Agent**：被测数字人，严格按任务指令扮演  
- **User 模拟器**：按分支 persona 扮演不同类型用户  

对话通过 SSE 实时流式展示，支持逐分支运行，全局互斥保证模拟稳定性。

### Step 4 · 评测报告

对话结束后自动评分（约 10–30 秒），报告包含：

- **综合分** + 五维雷达图  
- **优势 / 改进建议**（LLM 总结）  
- **维度详情**：每个子项的分数 + 中文理由（可展开审计）  
- **完整对话记录** 与 **原始任务指令** 对照查看

## 四、产品独特优势（评委速览）

### ✅ 优势 1：指令无关的统一评测框架

不需要为每个业务、每条 SOP 单独开发评测脚本。粘贴指令 → 自动生成 rubric → 统一五维输出。**降低 90% 以上的评测接入成本**（从「写 rubric」变为「贴指令」）。

### ✅ 优势 2：复杂语义可拆解、可评分

条件分支、FAQ、占位符、并列策略——通过 `item_kind` 分类 + 分支感知 + 三通道评分分别处理，解决「该评的不评、不该评的乱扣」行业痛点。

### ✅ 优势 3：结果可解释、可审计

每个评分子项附带具体中文理由，引用对话中的关键现象，而非黑盒总分。适合向业务方交付、向合规部门说明、向研发团队定位问题。

### ✅ 优势 4：多路径覆盖，接近真实上线场景

自动生成配合/拒绝/质疑/终止等分支，条件项只在对应分支生效。一次评测 ≈ 多轮人工 role-play 的自动化替代。

### ✅ 优势 5：端到端闭环，分钟级出报告

解析 → 模拟 → 打分 → 报告全自动，无数据库依赖，轻量可部署，适合快速迭代与黑客松 Demo。

### ✅ 优势 6：横向可对比

固定维度权重（35% / 25% / 15% / 15% / 10%），不同 Agent 版本、不同 Prompt 迭代、不同指令场景，报告结构一致，便于建立质量门禁。

## 五、评委 Demo 指引（建议 5 分钟）

### 路线 A：展示「复杂指令 → 统一标准」

1. 打开系统，选择预置 **「美团外卖·飞毛腿骑手通知」**  
2. 点击 **「解析指令」**，进入分支页 —— 指出系统自动生成了配合型/拒绝型等分支和完整评分维度  
3. 运行 **「拒绝型」** 分支，观看流式对话  
4. 进入报告页 —— 展示五维雷达图，展开 **「分支处理」** 维度：  
   - 指出「挽留」类子项有具体分数与理由  
   - 指出「鼓励能配送」类子项显示 **「场景未触发，不适用」** —— 体现系统**不会误扣**  
5. 展开 **「任务完成度」** —— 指出主动步骤若 Agent 漏做会低分 —— 体现系统**不会漏评**

### 路线 B：展示「格式不统一也能评」

1. 切换预置 **「课程平台·直播升级客服」**（Conversation Flow 格式，与路线 A 完全不同）  
2. 解析后对比：尽管指令结构不同，报告页仍是**同一套五维雷达**  
3. 运行 **「开车型」** 或 **「质疑型」** 分支，展示特殊终止/超范围场景的处理评分  

### 关键话术（可直接使用）

> 「业务方的任务指令各写各的，但 ConvoMatrix 把它们都翻译成同一套五维标准。  
> 条件分支不会误扣，主动步骤不会漏评，FAQ 不会在用户没问时被扣分——  
> 这就是我们把『复杂、不规则』变成『可度量、可对比』的核心能力。」

## 六、技术实现概要

| 模块 | 实现 | 作用 |
|------|------|------|
| 指令解析器 | `prompts/parser.py` | 指令 → 分支 + 五维 scoring_criteria |
| 后处理归一化 | `scoring/criteria_normalize.py` | 修正误分类、拆并列策略、FAQ/占位符语义化 |
| 对话模拟 | `routes/simulate.py` (SSE) | Agent + User 双 LLM 流式对话 |
| 规则评分 | `scoring/rules.py` | 字数、禁止词、重复、开场 — 确定性 |
| 语义评分 | `scoring/llm_scorer.py` + `prompts/scorer.py` | 按 item_kind 分流评分逻辑 |
| 聚合报告 | `scoring/aggregator.py` | 维度均分 + 效率 + 优劣势总结 |
| 前端 | React + Zustand + Recharts | 配置 / 分支 / 模拟 / 报告四页闭环 |

- **LLM**：DeepSeek `deepseek-chat`  
- **架构**：前后端分离，无状态设计，Docker Compose 一键部署  
- **并发**：LLM 评分 Semaphore(8)，流式 SSE 实时展示

## 九、总结

ConvoMatrix（DialogEval）面向对话式 AI 评测中最难的一环——**如何把千奇百怪的业务指令，变成公平、统一、可解释的质量标准**。

我们的独特优势可以概括为：

> **「指令可以不规则，但评分必须统一；逻辑可以很复杂，但结果必须可解释。」**

通过 **指令解析 → 语义分类 → 多分支模拟 → 三通道评分 → 五维报告** 的完整闭环，ConvoMatrix 让「数字人能不能把事办成」从主观感受，变为可度量、可对比、可迭代的工程能力。
